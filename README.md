# نظام تقييم استماع أغاني سونو اللامركزي v4 (Suno Music Rating System)

نظام متكامل، خفيف الوزن ولا مركزي (Decoupled Monorepo) مصمم للاستماع المقارن والتقييم النقدي الفني بين نسختين توليديتين (A و B) لمقاطع صوتية تم توليدها بواسطة ذكاء Suno الاصطناعي بناءً على قصائد شعرية وأناشيد دواوين.

---

## 🏗️ معمارية وهيكلية المشروع (Architecture Overview)

المشروع مبني بهيكلية **Monorepo** مفصولة بالكامل لضمان الخفة والتوافقية العالية مع أنظمة التخزين المسطحة وأنظمة الذكاء الاصطناعي (LLMs):

1. **الواجهة الخلفية (Backend - Flask):**
   - خادم REST API مبني بـ Python Flask.
   - تخزين مسطح ولا مركزي بالكامل على القرص بصيغة JSON (Flat-file JSON Storage)، لا يتطلب قواعد بيانات معقدة ويحفظ كل مشروع في مجلد خاص به: `backend/projects/<projectId>/`.
   - يتضمن نظام تحقق صارم من سلامة المشاريع الجديدة عبر مخطط هيكلي قياسي (JSON Schema).
   - يتعامل مع عمليات رفع ملفات الصوت، مزامنة التقييمات التلقائية، وخدمة الأصول الثابتة.

2. **الواجهة الأمامية (Frontend - React + Vite):**
   - تطبيق أحادي الصفحة (React SPA) حديث مبني بـ Vite و Tailwind CSS و Lucide Icons.
   - معتمد كلياً على مخرجات الخادم بمرونة برمجية كاملة، خالي تماماً من الأكواد الصلبة (Zero Hardcoded Configuration).
   - يتضمن منصة استماع ومقارنة زمنية متزامنة (Synchronized Time Listening Deck) للقفز التلقائي بين النسختين (A و B) في نفس اللحظة لسماع الفروقات الصوتية بدقة.
   - يدعم استيراد وتصدير حالات العمل كملفات التقييم الخام JSON، ونسخ التقارير الإحصائية الفنية المتكاملة بصيغة Markdown بضغطة زر.

## 🛠️ دليل البدء السريع وإعداد التطوير (Local Setup Guide)

### 1. تشغيل خادم الواجهة الخلفية (Backend Setup)

- **المتطلبات الأساسية:** Python 3.10+
- **خطوات التشغيل:**

  ```bash
  # 1. انتقل لمجلد الباكيند
  cd backend

  # 2. أنشئ البيئة الافتراضية وقم بتفعيلها
  python -m venv .venv
  # على Windows PowerShell:
  .venv\Scripts\Activate.ps1
  # على Unix/macOS:
  source .venv/bin/activate

  # 3. ثبت التبعيات اللازمة للفحص والتشغيل
  pip install -r requirements.txt pytest

  # 4. شغّل خادم المطور (يعمل افتراضياً على المنفذ 5000)
  python app.py
  ```

### 2. تشغيل الواجهة الأمامية (Frontend Setup)

- **المتطلبات الأساسية:** Node.js 18+
- **خطوات التشغيل:**

  ```bash
  # 1. انتقل لمجلد الفرونتيند
  cd frontend

  # 2. ثبت الحزم البرمجية النظيفة
  npm install

  # 3. شغّل خادم المطور (Vite Proxy سيقوم تلقائياً بتوجيه طلبات /api إلى خادم بايثون على المنفذ 5000)
  npm run dev
  ```

  _افتح المتصفح على الرابط: `http://localhost:3000`_

### 3. بناء نسخة الإنتاج والتشغيل المستقل (Production Build)

يقوم خادم Flask تلقائياً بتقديم ملفات الفرونتيند الثابتة إذا جرى بناؤها، مما يتيح تشغيل المشروع بالكامل كخدمة واحدة مدمجة على جهازك:

```bash
# داخل مجلد frontend، قم بالبناء النهائي:
npm run build
```

بمجرد انتهاء البناء بنجاح، يمكنك تشغيل خادم الباكيند فقط `python app.py` وزيارة الرابط `http://localhost:5000` وسيقوم خادم بايثون بخدمة الواجهة الأمامية والباكيند معاً وبشكل مستقل تماماً!

## 🧠 دليل الـ JSON Schema ونظام توليد المشاريع عبر الذكاء الاصطناعي (LLM project.json Spec)

المشروع مصمم ليكون لامركزياً بالكامل ومستقلاً عن المحتوى الصلب. يمكن لأي نموذج ذكاء اصطناعي (LLM) توليد ملف إعدادات متكامل للقصيدة باسم `project.json` ورفعه من الواجهة الأمامية مباشرة لبدء التقييم بشكل آلي.

لكي يقبل النظام ملف المشروع الجديد، يجب أن يتوافق بدقة مع مخطط الـ **JSON Schema** التالي (تطبق دالة `validate_project` في `schema.py` هذه القواعد):

### 1. المخطط الهيكلي القياسي للمشروع (Schema Spec)

يتكون ملف `project.json` من الحقول الأساسية التالية:

| الحقل              | النوع     | الوصف                                                              | إلزامي؟ |
| :----------------- | :-------- | :----------------------------------------------------------------- | :------ |
| `schemaVersion`    | `integer` | رقم إصدار المخطط الهيكلي (يجب أن يكون `1`)                         | نعم     |
| `projectId`        | `string`  | معرف فريد بالإنجليزية والأرقام فقط بدون مسافات `^[a-zA-Z0-9_-]+$`  | نعم     |
| `title`            | `string`  | العنوان الرئيسي للمشروع (مثال: "ديوان تقييم النابغة الذبياني")     | نعم     |
| `subtitle`         | `string`  | عنوان فرعي فني يصف المعايير ومحتوى التوليد                         | لا      |
| `poemTitle`        | `string`  | اسم القصيدة أو العمل الفني الأساسي                                 | لا      |
| `brandLetter`      | `string`  | الحرف المميز الذي يظهر كأيقونة دائرية في الهيدر (مثال: "ن")        | لا      |
| `footerNote`       | `string`  | نص تذييل الصفحة الفني                                              | لا      |
| `ratingScale`      | `object`  | مقياس التقييم المعتمد للتاقات، يحتوي على `id` ومصفوفة `options`    | نعم     |
| `tagTypes`         | `object`  | تصنيف التاقات وتسمياتها الفنية التي يعتمد عليها فحص الكود          | نعم     |
| `sectionQuestions` | `array`   | الأسئلة التقنية المشتركة التي سيتم سؤالها لكل الأقسام تلقائياً     | نعم     |
| `sections`         | `array`   | مصفوفة تفصيلية تحتوي على بيانات كل قسم والتاقات الخاصة به والكلمات | نعم     |

---

### 2. المواصفات البرمجية التفصيلية للحقول الفرعية (Detailed Sub-Objects)

#### أ. مقياس التقييم `ratingScale`

يحدد الخيارات المتاحة للمقيم لوضع علامات ملونة للتاقات الفنية.

```json
"ratingScale": {
  "id": "default_scale",
  "options": [
    { "value": "RED", "icon": "🔴" },
    { "value": "YELLOW", "icon": "🟡" },
    { "value": "GREEN", "icon": "🟢" }
  ]
}
```

_ملاحظة: برمجياً، الاصطلاح الفني يقتضي أن الخيار الأسوأ (Worst Rating) يجب رصفه كأول عنصر في مصفوفة الخيارات (وهو RED في المثال أعلاه)، حيث يعتمد عليه نظام SummaryDashboard لحساب التاقات التي تحتاج لمراجعة فنية طارئة تلقائياً._

#### ب. تصنيف التاقات `tagTypes`

القاموس الحاكم لتصنيف التاقات الهندسية وتوضيح معانيها للمقيم بلغة مفهومة:

```json
"tagTypes": {
  "melody": {
    "label": "اللحن والانسجام الملحي",
    "ratingLabels": {
      "RED": "خروج عن المقام / نشاز",
      "YELLOW": "مقبول تقنياً",
      "GREEN": "انسجام رائع وتدفق ممتاز"
    }
  },
  "vocal": {
    "label": "جودة وهندسة الصوت",
    "ratingLabels": {
      "RED": "ضوضاء / تداخل تآلفات",
      "YELLOW": "صوت بشري مقبول",
      "GREEN": "هندسة نظيفة ومتوازنة"
    }
  }
}
```

#### جـ. الأسئلة المشتركة للأقسام `sectionQuestions`

توليد أسئلة تظهر في كل الأقسام تلقائياً لقياس العيوب المتكررة:

```json
"sectionQuestions": [
  {
    "id": "drift",
    "label": "الهروب النغمي والانتقال المفاجئ (Genre Drift)؟",
    "type": "single_choice",
    "options": ["no", "yes"],
    "optionLabels": { "no": "لا يوجد، انتقال سلس", "yes": "نعم، حدث هروب مفاجئ" },
    "hasDetails": true,
    "detailsLabel": "سجل توقيت ونوع الهروب النغمي بالثواني:"
  }
]
```

_أنواع الأسئلة المتاحة:_ `single_choice` (اختيار مفرد راديو)، `boolean` (نعم/لا)، `free_text` (إجابة حرة).

#### د. مواصفات الأقسام `sections`

كل قسم يمثل مقطعاً طويلاً أو شطراً من القصيدة، ويتضمن التاقات المخصصة للفحص البرمجي:

```json
"sections": [
  {
    "id": 1,
    "title": "القسم الأول — فاتحة القصيدة",
    "meta": { "maqam": "حجاز", "tempo": "ثابت" },
    "description": "الأبيات الشعرية الأولى التي تقدم روح العمل الغنائي الفني",
    "watchpoint": "راقب تدفق الحروف ونطق الكلمات العربية المخارجية بوضوح صريح.",
    "specialQuestion": "هل نجح الذكاء الاصطناعي في تمثيل حزن المقام؟",
    "lyrics": "أَمِنْ آلِ مَيَّةَ رَائِحٌ أَوْ مُغْتَدِي ... عَجْلانَ ذَا زَادٍ وَغَيْرَ مُزَوَّدِ",
    "tracks": {
      "A": {
        "assigned_filename": "1_A.mp3",
        "styles": "Classic Arabic Maqam, Hijaz style, slow oud, solo vocal",
        "clip_id": "suno_clip_id_for_version_a"
      },
      "B": {
        "assigned_filename": "1_B.mp3",
        "styles": "Traditional Arabesque Oud, slow rhythm, vocal clarity",
        "clip_id": "suno_clip_id_for_version_b"
      }
    },
    "tags": [
      {
        "id": "sec1_tag1",
        "tag": "vocal_clarity",
        "expected": "مخارج الحروف ونطق الميم والهاء بوضوح كامل دون دمج صوتي سيئ",
        "type": "vocal"
      }
    ]
  }
]
```
